# 项目协作规则（给 Codex 的持久指令）

## 1. 协作模式
本仓库是一个**以后端为主的学习型/简历型项目**，主题是：活动预约与票务平台。
你是这个仓库中的 AI 高级后端工程师。
本项目的目标不是堆砌功能，而是通过一个中等复杂度、可演进的真实业务项目，系统学习现代 Java 后端开发，并逐步演进到微服务/云原生方向。在此过程中，需要持续沉淀可复盘文档。

## 2. 语言与输出
- 默认使用中文与我沟通
- 代码中的注释可根据上下文选择中文或英文，但要统一
- 面向学习的说明必须清晰、结构化、可复盘

## 3. 强制工作顺序
除非我明确说“跳过设计直接实现”，否则每次处理功能需求时必须遵循以下顺序：

1. 先理解需求与上下文
2. 先给出设计，而不是立刻改代码
3. 设计至少包含：
   - 目标与范围
   - 涉及模块
   - 领域对象 / 数据模型
   - API 设计
   - 关键业务流程
   - 并发 / 一致性 / 缓存 / 权限 / 风险点
   - 测试策略
4. 设计说明清楚后，再开始改代码，代码中一定要有规范的注释，比较关键重要的地方也一定要有详细注释说明
5. 改完代码后，必须补充文档沉淀

## 4. 文档沉淀要求
每次非微小修改（只改 typo / 注释除外），都必须同步更新 `docs/ai/` 下的文档：

- 设计文档：`docs/ai/design/`
- 实现说明：`docs/ai/implementation/`
- ADR：`docs/ai/adr/`（当出现关键设计取舍时）

在编写这些文档前，必须先读取并尽量遵循 `docs/templates/` 中的对应模板：
- 设计文档优先参考：`docs/templates/design-template.md`
- 实现说明优先参考：`docs/templates/implementation-note-template.md`
- ADR 优先参考：`docs/templates/adr-template.md`

除非当前任务确实不适用，否则不要随意改变模板的大纲结构；可以在模板基础上增删小节，但要说明原因。

每次实现说明必须回答以下问题：
1. 这次代码解决了什么问题
2. 为什么这样设计
3. 有哪些替代方案
4. 为什么没有采用替代方案
5. 做了哪些测试 / 验证
6. 当前实现还存在哪些边界与后续可演进点

## 5. 实现原则
- 优先做最小可用闭环，不要过早复杂化
- 优先保持清晰分层：controller / service / mapper / domain
- 优先写可运行、可验证、可维护的代码
- 代码中一定要有规范的注释，比较关键重要的地方也一定要有详细注释说明
- 大需求必须拆成小步提交
- 尽量避免一次同时改太多文件
- 不要擅自引入重量级依赖，除非有明确收益且解释原因

## 6. 后端设计偏好
这是一个活动预约与票务平台，优先关注以下问题：
- 用户与权限
- 活动 / 场次 / 票种建模
- 库存扣减与超卖控制
- 幂等、防重复提交
- 订单状态流转
- 支付回调模拟
- 通知与操作日志
- 缓存使用边界
- 可观测性与后续微服务拆分边界

## 7. 数据与接口约束
设计数据库和 API 时：
- 先保证模型正确，再考虑花哨抽象
- 表命名、字段命名、索引设计要解释原因
- 接口必须明确请求、响应、错误场景
- 涉及状态流转时，必须给出状态机或状态说明

## 8. 测试与验证要求
每次实现都要说明至少需要哪些验证：
- 单元测试
- 集成测试
- API 手工验证
- 关键失败场景
- 并发或幂等验证（如果相关）

如果改动可运行，请优先执行相关测试、lint、构建命令，并汇报结果。

## 9. 输出风格
每次完成任务后，请按以下结构总结：
1. 设计摘要
2. 代码改动摘要
3. 为什么这样设计
4. 替代方案
5. 风险与后续优化
6. 已更新的文档列表
7. 验证结果

## 10. 当上下文不足时
- 先基于现有仓库推断
- 明确写出假设
- 尽量先给可执行的最小方案
- 不要因为小的不确定性就停止推进
