## 3. 30 秒项目介绍

这个项目是一个活动预约与票务平台后端，我在阶段 1 完成了账号体系和权限基础设施。用户可以注册、登录并获取 JWT，后续请求通过 Bearer token 建立当前用户上下文。权限上采用 USER / ADMIN 的 RBAC 模型，管理员接口同时有 URL 级和方法级保护。比较关键的设计是 JWT 只保存用户 ID，不保存角色快照，每次请求都会重新查询用户状态和角色，所以用户被禁用后旧 token 也会被拒绝。

## 4. STAR 示例

背景：

项目后续会涉及活动管理、订单创建、支付回调和操作日志，这些业务都需要知道当前请求用户是谁，以及用户是否有权限访问目标资源。因此阶段 1 必须先完成认证、登录态和基础权限边界。

任务：

我负责设计并实现注册登录、JWT access token、当前用户接口、USER / ADMIN RBAC、管理员用户接口和安全异常处理，同时保证这套能力能被后续业务模块复用。

行动：

- 建立 `users / roles / user_roles` 三张表，并初始化 USER、ADMIN 角色和本地管理员账号。
- 用 BCrypt 保存密码哈希，注册时在同一事务中创建用户并绑定默认 USER 角色。
- 使用 Spring Security 接入自定义 JWT Filter，登录成功后签发 access token。
- 将 JWT claims 收缩到最小主体 ID，认证时通过用户 ID 查询最新用户状态和角色。
- 将安全基础设施拆到 `infra.security`，将账号业务和权限加载留在 `modules.auth`。
- 使用 `hasRole("ADMIN")` 和 `@PreAuthorize` 保护管理员接口。
- 为管理员用户列表补充分页、筛选和批量角色查询，避免全量查询和 N+1。
- 编写集成测试覆盖注册、登录、401、403、过期 token、篡改 token、禁用用户旧 token 和管理员查询等场景。

结果：

阶段 1 形成了可演示的安全闭环：普通用户可以注册登录并访问当前用户接口，USER 访问管理员接口会返回 403，禁用用户不能登录且旧 token 会在下一次请求失效，管理员可以分页筛选用户并更新用户状态。这套当前用户上下文后续可以直接支撑活动管理、订单归属、支付审计和操作日志。

## 5. 常见面试追问

### JWT 里放了什么？为什么不放 roles？

当前 JWT 只放最小身份信息：`sub` 对应 `users.id`，以及 `iss`、`iat`、`exp`。不把 roles 作为最终授权依据写入 token，是因为角色和用户状态会变化。如果 token 内保存角色快照，用户被禁用或角色被收回后，旧 token 在过期前仍可能带着旧权限访问系统。当前做法是每次请求查库加载最新状态和角色，安全性更清晰。

### 如何保证禁用用户旧 token 失效？

JWT 验签通过后不会直接放行。`JwtAuthenticationFilter` 会解析出用户 ID，再调用 `AuthenticatedPrincipalService` 查询用户表。如果用户不存在或状态为 `DISABLED`，认证失败并返回 401。因此禁用操作会在该用户下一次请求时生效，不需要等待 token 自然过期。

### 401 和 403 怎么区分？

401 表示认证失败，常见原因是未携带 token、token 过期、签名非法、issuer 不匹配、用户不存在或用户被禁用。403 表示已经认证成功，但权限不足，例如普通 USER 访问 `/api/v1/admin/users`。

### 为什么 logout 不做服务端吊销？

当前阶段只实现无状态 access token，服务端不保存 token，所以 logout 是 no-op 语义，代表客户端应删除本地 token。要实现服务端主动吊销，需要引入 refresh token、token id、Redis 黑名单、登录会话表或 token version，这些会带来存储、过期清理和并发刷新问题，阶段 1 先不提前复杂化。

### 注册接口如何处理并发重复账号？

服务层先检查 username 和 email 是否已存在，用于返回友好错误。但真正保证并发正确性的是数据库唯一约束。两个请求同时通过预检查时，只有一个能插入成功，另一个会触发唯一键异常，再被转换为稳定业务错误。

### 为什么只做 USER / ADMIN，不做更细的权限点？

阶段 1 的核心是建立认证和管理端边界。活动、订单、票种等资源模型还没展开时，提前设计权限点和数据范围容易空转。当前 `roles` 和 `user_roles` 已经为角色扩展留出空间，等后续资源稳定后再引入 permissions 或资源级授权更合理。

### Spring Security 是怎么保护管理员接口的？

有两层：`SecurityConfig` 在 URL 级对 `/api/v1/admin/**` 配置 `hasRole("ADMIN")`，进入 Controller 前就能拦截；`AdminUserController` 再用 `@PreAuthorize("hasRole('ADMIN')")` 做方法级保护。角色编码从数据库读出后会转换成 `ROLE_USER`、`ROLE_ADMIN` 这类 Spring Security authority。

### 为什么公开接口要显式写 HTTP Method？

Spring Security 中不写 Method 的 matcher 会匹配所有方法。公开接口如果写成 `/api/v1/system/**.permitAll()`，未来同一路径下新增接口可能被意外公开。现在只对真实存在且需要公开的 `GET /ping`、`POST /echo` 等接口按方法放行，公开面更小。

### 管理员用户列表有什么设计点？

它不是简单 `findAll`。当前支持分页、账号字段筛选、状态筛选、时间范围筛选，默认按 `created_at DESC, id DESC` 展示新注册用户，并且按当前页用户 ID 批量查角色，避免每个用户单独查一次角色造成 N+1。

### 这个阶段还有哪些风险？

主要风险包括：无状态 logout 不能吊销旧 token；每次请求查库加载用户和角色会增加数据库压力；权限模型只有角色级；管理员种子账号不能用于生产；管理员列表在大数据量下需要补索引和搜索策略。后续可以引入 refresh token、token version、权限缓存、权限点模型和登录审计。

## 6. 可量化表达方向

这个阶段适合强调安全闭环和测试覆盖，不适合夸大性能指标：

- 覆盖注册、登录、当前用户、登出语义、管理员列表和用户状态更新 6 类认证授权接口。
- 覆盖 USER / ADMIN 两类角色边界。
- 覆盖未登录、权限不足、过期 token、篡改 token、禁用用户旧 token 等认证授权失败场景。
- 将管理员用户列表从全量查询演进为分页、筛选、排序和批量角色查询。
- 生产 profile 下默认关闭 OpenAPI 与 Swagger UI，减少接口契约暴露面。
