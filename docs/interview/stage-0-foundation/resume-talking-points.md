# 阶段 0 简历表达

## 1. 一句话项目介绍

EventHub 是一个活动预约与票务平台后端项目，采用 Spring Boot 单体起步，围绕用户权限、活动票务、订单库存、支付回调和通知审计逐步演进；阶段 0 先完成统一响应、异常治理、请求追踪、数据库迁移、接口文档、健康检查和容器化本地环境等工程底座。

## 2. 简历 bullet 备选

- 从 0 搭建 Spring Boot 后端基础工程，设计 `common / infra / modules` 分层，完成统一响应、统一异常、参数校验和错误码治理，为后续用户、活动、订单等业务模块提供稳定接口契约。
- 引入 requestId 请求追踪机制，通过 Filter 生成和校验请求标识，并在 HTTP 出口层为统一响应体补充 requestId，实现响应头、响应体和日志上下文的一致关联。
- 接入 Flyway、MySQL、Redis、OpenAPI 和 Actuator，提供 `dev / test / prod` 分环境配置，支持健康检查、接口文档和数据库基线迁移。
- 编写多阶段 Dockerfile，并通过 Docker Compose 编排 MySQL、Redis 和后端应用，实现本地一键启动的后端基础闭环。
- 补充单元测试和 Spring Boot 集成测试，覆盖成功响应、参数校验失败、非法 JSON、资源不存在、未知异常兜底和 requestId 透传等基础场景。

## 3. STAR 示例

背景：

项目是一个用于学习和简历沉淀的活动预约与票务平台，后续会涉及注册登录、RBAC、活动管理、库存扣减、订单状态流转和支付回调。为了避免后续业务阶段反复返工基础设施，需要先建立可靠的后端工程底座。

任务：

我负责阶段 0 的基础工程搭建，目标是让项目具备可运行、可测试、可本地复现的最小闭环，并沉淀统一的接口协议、异常处理、请求追踪和容器化启动方式。

行动：

- 采用 Spring Boot 单体单模块起步，按 `common / infra / modules` 拆分基础职责。
- 设计统一响应体和错误码，将参数校验、业务异常、资源不存在和未知异常收敛到全局异常处理器。
- 通过 `RequestIdFilter` 和 `ResponseBodyAdvice` 拆分 requestId 生命周期管理与响应体注入，避免通用响应模型依赖 MDC。
- 使用 Flyway 建立数据库基线，接入 MySQL、Redis、OpenAPI、Actuator。
- 提供 Dockerfile 和 Docker Compose，让 MySQL、Redis 和后端应用可以一键启动。
- 使用 H2 和 MockMvc 补充基础测试，验证接口、异常、requestId 和健康检查链路。

结果：

阶段 0 形成了完整基础闭环：本地可以通过 Compose 启动依赖和应用，接口文档和健康检查可访问，统一响应和异常处理有测试覆盖，后续阶段可以直接在这个底座上继续实现注册登录、权限、活动、订单和支付模块。

## 4. 常见面试追问

### 为什么一开始没有做微服务？

因为阶段 0 的核心目标是建立可运行、可验证的工程底座。业务边界还没有稳定时直接拆微服务，会把复杂度提前放到服务治理、部署和调用链上，反而影响业务建模。当前采用单体单模块起步，但通过 `modules` 目录保留业务边界，后续可以先演进为多模块单体，再评估微服务拆分。

### 统一响应体为什么不直接读取 MDC？

`ApiResponse` 属于通用响应协议模型，如果它直接读取 MDC，就会让 common 层隐式依赖日志上下文，也会影响异步任务、定时任务等非 HTTP 场景。当前做法是 Filter 负责 requestId 生命周期，HTTP 出口层 Advice 负责给 `ApiResponse` 补充 requestId，职责更清晰。

### 为什么 Actuator 和 OpenAPI 不包装成统一响应？

这些端点属于框架和工具生态的一部分，调用方通常期望它们保持原生格式。如果强行包装，会影响健康检查、接口文档工具和后续平台探针。因此统一响应只约束业务 API，Actuator 与 OpenAPI 保持原生响应。

### 为什么测试环境用 H2？

阶段 0 的测试重点是验证工程链路，包括 Controller、校验、异常、统一响应和 requestId。H2 可以让测试不依赖外部容器，更适合快速运行。但它不能完全替代 MySQL，后续涉及复杂 SQL、事务、锁、库存扣减时，需要补真实 MySQL 集成测试。

### Docker Compose 为什么默认启动后端应用？

因为阶段 0 的验收目标是本地一键启动完整基础闭环。默认启动后端可以降低新环境复现和演示成本；日常开发仍然可以只启动 MySQL/Redis，再用 IDE 或 Maven 运行后端。

### 这个阶段最能体现什么能力？

不是业务功能复杂度，而是工程化意识：知道一个后端项目在写业务前需要统一接口契约、异常治理、请求追踪、配置分层、数据库迁移、接口文档、健康检查、测试和容器化运行方式。

## 5. 可量化表达方向

当前阶段更适合强调“覆盖面”和“闭环”，而不是夸大性能指标：

- 覆盖统一响应、异常处理、参数校验、requestId、OpenAPI、Actuator、Flyway、Docker Compose 等基础能力。
- 覆盖成功响应、校验失败、非法 JSON、404、未知异常、requestId 注入和非统一响应不包装等测试场景。
- 支持一条 Compose 命令启动 MySQL、Redis 和后端应用。

## 6. 后续衔接阶段 1

阶段 1 的注册登录、JWT 与 RBAC 可以自然承接阶段 0：

- 安全错误可以复用统一错误码和响应体。
- 登录、鉴权、权限失败可以继续使用 requestId 排查。
- 用户与角色模块可以沿用 `modules` 分层。
- OpenAPI 和测试策略可以继续沉淀为阶段级可展示成果。
