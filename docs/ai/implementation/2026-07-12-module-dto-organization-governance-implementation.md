# 模块 DTO 组织规则落地实现说明

> 本文只记录设计、规则和执行入口的落地，不表示 Java 包迁移已经完成。

## 1. 本次改动解决了什么问题

- 将“移除响应 `vo`、统一使用请求/响应 DTO”的口头方案沉淀为可复盘的设计文档和 accepted ADR。
- 修正同级 `dto/query`、`dto/request` 分类重叠的问题，统一为 `dto/request`、`dto/response`。
- 把约定写入 `AGENTS.md`，使后续活动、场次、票种、订单、支付、通知等模块都必须遵循同一组织原则。
- 在功能开发和代码复盘提示词中加入 DTO 目录与边界检查，降低规则只存在于单篇文档、后续开发遗忘的风险。
- 提供一份可直接交给其他智能体执行的迁移提示词，冻结当前类型映射、范围、非目标和验证标准。
- 明确当前代码仍未迁移，避免把规则落地误写为功能实现完成。

## 3. 为什么这样设计

- `AGENTS.md` 是仓库内所有后续智能体都会读取的持久规则入口，只写 ADR 不能保证新模块开发时主动执行。
- ADR 负责记录长期决策及备选方案，设计文档负责记录当前代码的精确迁移方案，两者职责不同。
- 功能提示词负责在开发入口提醒规则，review 提示词负责在完成后检查规则，形成“开发前约束 + 开发后审计”的闭环。
- 当前代码尚未迁移，因此 README、路线图和面试材料仍应描述真实代码现状；提前把它们改成目标目录会造成文档与代码不一致。
- 历史 `docs/ai` 文档记录过去真实选择，不做机械全局替换。新 ADR 清楚声明旧响应 `vo` 约定已被替代，既保留演进历史，也建立新的有效基线。
- 专用执行提示词直接引用设计与 ADR，并给出完成定义，可以让后续智能体聚焦执行而不是重复讨论已经批准的分类方案。

## 4. 替代方案

- 方案 A：只新增一份设计文档，不修改 `AGENTS.md`。
  - 未采用，因为后续智能体不一定主动找到该文档，无法形成持久规则。
- 方案 B：现在直接同步 README 和路线图中的目录树。
  - 未采用，因为 Java 代码还没有迁移，当前状态文档会先于实现而失真。
- 方案 C：直接修改所有历史设计、实现和 ADR 中的 `DTO/VO` 术语。
  - 未采用，因为会抹去项目真实演进过程，也容易把当时正确的实现记录改成从未发生过的状态。
- 方案 D：只在对话中给出提示词，不纳入仓库。
  - 未采用，因为仓库文件更容易版本化、复用和由后续智能体直接读取。
- 方案 E：本轮顺便完成 Java 包迁移。
  - 未采用，因为用户明确要求只沉淀文档和规则，将代码实现交给另一个智能体。

## 5. 测试与验证

- 跑了哪些测试
  - 本轮没有修改 Java、配置或运行时资源，因此没有运行 Maven 测试。
- 手工验证了哪些场景
  - 检查设计文档保持 `docs/templates/design-template.md` 的 1-12 章节顺序。
  - 检查 ADR 保持 `docs/templates/adr-template.md` 的标题、状态、背景、决策、备选方案、理由和影响结构。
  - 检查本实现说明回答了问题、改动、原因、替代方案、验证、限制和后续影响。
  - 检查 `AGENTS.md` 已将规则设为所有新模块和现有模块后续调整的强制约束。
  - 检查执行提示词明确了不保留兼容层、精确类型映射、`TokenService` 边界调整、文档同步和完成定义。
- 结果如何

  - Java 迁移仍为待办，未将当前代码状态误报为完成。

## 6. 已知限制

- `auth`、`system` 仍存在 `vo` 包和旧类型名；这是已记录、待后续智能体执行的存量迁移。
- README、阶段 0/1 路线图和面试材料仍显示当前旧目录，只有代码迁移完成后才能同步更新。
- 本轮没有执行编译、单元测试或集成测试，因为没有运行时代码改动。
- 规则依赖后续开发和 review 按 `AGENTS.md` 执行；如果未来引入非 Codex 开发流程，还可以补充 ArchUnit 或自定义静态结构测试进行机器约束。
- 当前不引入 ArchUnit，避免仅为 5 个存量响应类型增加依赖；先通过规则、静态搜索和测试完成最小闭环。

## 7. 对后续版本的影响

- 对简历可用版的价值
  - 展示项目不仅关注功能实现，也能识别 DTO/VO 概念重叠并通过 ADR、规则和 review 机制完成工程治理。
  - 后续模块目录更统一，接口输入、接口输出、持久化模型和领域模型的边界更容易讲清楚。
- 对微服务 / 云原生演进的影响
  - 清晰的请求/响应传输边界有利于未来拆分服务契约或迁移到 OpenAPI First。
  - 领域值对象不再与响应 `vo` 混淆，为后续复杂领域建模保留空间。
  - 如果未来服务入口不再只有 HTTP，可以在不破坏当前 DTO 规则的前提下增加独立应用 command/query 模型。
