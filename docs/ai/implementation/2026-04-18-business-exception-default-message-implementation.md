# BusinessException 默认文案构造实现说明

## 1. 本次改动解决了什么问题

`BusinessException` 之前只能通过 `(ErrorCode, String)` 构造。当业务层希望直接使用 `ErrorCode` 中定义的默认错误描述时，需要重复传入 `errorCode.getDefaultMessage()`，不够简洁，也增加了文案不一致的风险。

## 2. 改动内容
- 新增了 `BusinessException(ErrorCode errorCode)` 构造方法，默认使用错误码中的默认描述
- 修改了现有构造逻辑，为两个构造方法统一增加 `errorCode` 空值保护
- 新增了 `BusinessExceptionTest`，覆盖默认文案、自定义文案和空值保护场景

## 3. 为什么这样设计
- 保留原有 `(ErrorCode, String)` 构造方法，确保业务层仍能返回更精确的错误提示
- 新增便捷构造方法，减少重复样板代码，提升错误码默认文案的复用率
- 在构造阶段完成空值保护，避免进入全局异常处理时才暴露缺失错误码的问题

## 4. 替代方案
- 方案 A：只保留 `(ErrorCode, String)` 构造方法，由业务代码手动传 `defaultMessage`
- 方案 B：允许 `errorCode` 为空，在全局异常处理器中兜底映射成系统错误
- 为什么没有采用：
  - 不选方案 A，因为它会让默认文案复用退化成重复样板代码
  - 不选方案 B，因为业务异常失去错误码约束后，统一错误处理会变得更脆弱，也不利于快速定位问题

## 5. 测试与验证
- 新增单元测试 `BusinessExceptionTest`
- 验证了以下场景：
  - 只传 `ErrorCode` 时使用默认文案
  - 传入自定义文案时保留自定义文案
  - 两个构造方法都拒绝 `null` 错误码
- 运行验证命令：`cd backend && mvn test -Dtest=BusinessExceptionTest`

## 6. 已知限制
- 当前 `ErrorCode` 仍然比较粗粒度，默认文案更适合作为兜底提示，而不是覆盖所有业务提示场景
- 本次没有引入国际化文案或错误码分组机制

## 7. 对后续版本的影响
- 对简历可用版的价值：体现了统一错误处理体系的可维护性和可扩展性
- 对微服务 / 云原生演进的影响：异常对象契约更稳定，有利于后续在网关层、服务间调用或统一错误码治理中复用
