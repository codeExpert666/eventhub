# ADR：后端单体工程根目录布局

## 标题
仓库根目录直接作为单体后端 Maven 构建根

## 状态
- accepted

## 背景
EventHub 当前是以后端为主体的单体单模块 Spring Boot 项目。最初后端工程放在 `backend/` 下，为未来同仓多工程预留了物理边界；但目前仓库没有前端或其他 Maven 模块，这一层级使 Maven 命令、Docker Compose 构建上下文、IDE 导入和工具索引都需要额外维护 `backend/` 路径。

本决策只调整仓库布局，不推翻 `2026-04-16-backend-foundation-monolith.md` 中“单体单模块”的架构选择。

## 决策
- 仓库根目录同时作为当前唯一 Spring Boot 应用的 Maven 构建根。
- `pom.xml`、`Dockerfile`、`lombok.config` 和 `src/` 直接放在根目录，不再保留 `backend/` 容器目录。
- Docker Compose 的后端构建上下文使用仓库根目录，并通过 `.dockerignore` 排除非构建输入。
- `backend` 仍可作为 Maven artifactId、Compose 服务名和运行时应用角色名称；工程语义名称不因物理目录上移而强制改名。
- 既有 `docs/ai/` 文档保留旧路径以维持历史真实性，本 ADR 和本次设计/实现说明作为新目录约定的生效边界。

## 备选方案
- 方案 1：继续保留 `backend/` 子目录。
- 方案 2：根目录建立聚合父 POM，`backend/` 作为唯一 Maven 子模块。
- 方案 3：只移动部分构建文件，保留源码或 Dockerfile 在子目录中。

## 决策理由
- 当前只有一个后端应用，仓库根和模块根合并后命令最短、认知成本最低。
- 根目录直接执行 Maven 更符合主流单模块 Spring Boot 项目习惯，也减少 IDE、CI 和代码分析工具的额外工作目录配置。
- 通过 `.dockerignore` 可以控制根构建上下文，不需要依赖子目录来隔离本地元数据和文档。
- 保留 artifactId 和服务名可以避免无业务收益的产物命名、镜像命名和运行配置变更。
- 如果未来确实出现前端或多个独立部署单元，再基于真实边界演进为多模块或多工程布局，比当前提前引入聚合层更合适。

## 影响
- 好处：
  - Maven、Docker、IDE 和本地工具共享同一工程根。
  - README、测试命令和后续自动化不再需要 `cd backend`。
  - 减少路径配置和目录结构理解成本。
- 代价：
  - 仓库根目录的工程文件更多。
  - 未来加入前端或多个构建单元时，可能需要再次调整目录布局。
  - 历史文档和新目录约定会同时存在，检索路径时需要关注文档时间语境。
- 后续可能需要调整的地方：
  - 当出现第二个真实构建单元时，重新评估聚合 Maven、多模块单体或 `backend/` / `frontend/` 对称布局。
  - CI/CD 建立后，所有工作目录和缓存键应以仓库根 Maven 工程为准。
