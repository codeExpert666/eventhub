# ADR：采用受约束的 Lombok 使用规范

## 标题
采用受约束的 Lombok 使用规范治理后端样板代码

## 状态
- accepted

## 背景
当前后端项目处于基础工程阶段，已经有统一响应、统一异常、系统模块和 OpenAPI 基础链路。随着后续用户、活动、场次、票种、订单、库存、支付等模块陆续引入，构造器、getter、日志字段等样板代码会快速增加。

这些样板代码本身不承载核心业务语义，但会占据阅读空间。项目需要减少重复代码，同时避免 Lombok 被滥用后带来对象可变性扩大、调试不直观、初学者难以理解等问题。

## 决策
在 `backend` 模块引入 Lombok，但采用受约束的使用规范：

- 允许使用 `@RequiredArgsConstructor` 生成 `final` 依赖字段构造器，用于保持构造器注入。
- 允许使用 `@Getter` 生成只读字段访问器，用于错误码、异常等不可变对象。
- 允许使用 `@Slf4j` 生成日志字段，用于减少重复的 logger 声明。
- 默认不使用 `@Data`、`@Setter` 这类会扩大对象可变性的注解。
- DTO/VO 如果本质是只读数据载体，优先继续使用 Java `record`。
- Lombok 只作为编译期工具参与 annotation processing，并从 Spring Boot 可执行 Jar 中排除。

## 备选方案
- 方案 1：不引入 Lombok，继续全部手写样板代码。
- 方案 2：全面使用 Lombok，统一将 DTO/VO 和领域对象改为 Lombok POJO。
- 方案 3：仅加入 Lombok 依赖，不制定使用边界。

## 决策理由
- 选择受约束的 Lombok，可以减少低价值重复代码，又不会改变现有接口、异常和运行时行为。
- 构造器注入、只读 getter、日志字段是 Lombok 的低风险高收益场景，适合当前阶段使用。
- `record` 对简单只读 DTO/VO 的表达更直接，不应该为了工具统一而放弃语言本身的不可变建模能力。
- 不默认使用 `@Data` 和 `@Setter`，可以避免请求模型、响应模型和错误码对象被无意改成可变对象。
- 显式配置 annotation processor 和 Spring Boot 打包排除，能让编译期依赖和运行时依赖边界更清楚。

## 影响
- 好处
  - 减少后续模块中的构造器、getter 和 logger 样板代码。
  - 让业务类更聚焦领域规则、状态流转和边界条件。
  - 保持 Lombok 不进入运行时依赖，降低运行环境复杂度。
- 代价
  - 初学者需要理解 Lombok 的编译期生成机制。
  - IDE 需要正确启用 annotation processing。
  - 部分生成方法不再直接出现在源码中，需要依赖 IDE 或编译结果查看。
- 后续可能需要调整的地方
  - 当代码规模扩大后，可以引入 Checkstyle、ArchUnit 或自定义静态检查，限制 `@Data`、`@Setter` 的使用范围。
  - 如果未来出现领域对象复杂不变式，需要优先考虑显式构造方法或工厂方法，而不是机械套用 Lombok 注解。
