# 标题
基础阶段采用单体单模块 Spring Boot 工程，并通过 Docker Compose 提供 MySQL / Redis 本地依赖

## 状态
- accepted

## 背景
当前项目处于阶段 0，需要先把后端基础工程搭起来，为后续用户、活动、订单和支付模块提供稳定的目录结构、配置方式和本地运行环境。

此时有几个可选方向：
- 一开始就拆成多模块 Maven 工程
- 一开始就按微服务思路分多个应用
- 先做单体单模块基础工程，外接 MySQL 和 Redis

## 决策
选择单体单模块 Spring Boot 工程作为当前阶段的工程组织方式，并使用 Docker Compose 在本地提供 MySQL / Redis 依赖。

## 备选方案
- 方案 1：多模块 Maven 单体
- 方案 2：微服务骨架
- 方案 3：只使用内存数据库或不接入外部依赖

## 决策理由
选择当前方案的原因：

- 当前目标是“最小可用闭环”，单体单模块最容易先把基础工程跑通
- 后续业务模块还未展开，过早拆模块会增加构建和依赖管理复杂度
- 活动票务项目后续一定会用到 MySQL 和 Redis，因此应尽早把本地依赖环境和配置方式固定下来
- 通过 Docker Compose 管理依赖，比手工安装环境更容易复现，也更适合写进 README 和面试材料

## 影响
- 好处：
  - 起步成本低，便于快速进入业务建模阶段
  - 目录边界清晰，后续继续演进时不需要推倒重来
  - 本地环境可复现，适合学习、演示和面试表达
- 代价：
  - 当前没有多模块带来的物理隔离
  - 后续如果拆服务，需要重新梳理模块边界和共享能力
- 后续可能需要调整的地方：
  - 当用户、活动、订单模块逐渐复杂后，可以先考虑从单模块演进到多模块单体
  - 当异步通知、支付回调和可观测性需求增强后，再考虑服务拆分或事件驱动架构
